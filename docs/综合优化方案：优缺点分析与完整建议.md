综合优化方案：优缺点分析与完整建议
一、已提出各优化方案的优缺点对比
🔴 P0 类：修复关键 Bug
1. 修复 Score Threshold 逻辑（rag_chain.py:293）
优点	缺点
工作量极小（改一个符号）	修复后可能暴露原来被掩盖的其他问题（如真正的低质量文档涌入）
直接修复最严重的检索质量 Bug	需要重新评估默认阈值是否合理
零副作用	—
2. BM25 缓存持久化 + 失效机制
优点	缺点
消除每次搜索重建索引的 O(N) 开销	需要维护缓存状态，增加内存占用
修复新增文档后索引不更新的 bug	多进程/多实例环境下缓存一致性复杂
搜索延迟大幅下降	冷启动第一次仍需全量建索引
🟠 P1 类：接通已有功能
3. API 接入完整 RAGChain
优点	缺点
无需新增功能，让 hybrid search / parent-child / reranking 等已有代码真正生效	API 层需要重构，引入一定测试风险
收益巨大（目前 API 只用 similarity_search，是最低端的检索）	RAGChain 和 API 各自有 prompt 逻辑，需要统一合并
结构更清晰	需要处理 RAGChain 和 API 参数兼容性
4. BM25 加 jieba 分词 + 停用词过滤
优点	缺点
中文 BM25 检索质量大幅提升	引入新依赖（jieba ~10MB）
消除"是"、"的"、标点符号等无意义词的权重干扰	jieba 分词速度略慢，大批量文档处理时有影响
对英文文档同样有改善（NLTK 停用词）	需要维护停用词表，领域词可能误过滤
5. Prompt 统一增强
优点	缺点
统一 RAGChain 和 API 两处分散的 prompt 逻辑	prompt 修改难以自动测试，依赖主观评估
加入反幻觉指令、引用要求后生成质量提升	不同模型（本地 Ollama vs API）对同一 prompt 响应差异大
加入意图感知后不同类型问题可差异化处理	Prompt 过长会占用 context window，影响生成
🟡 P2 类：新增能力
6. 查询预处理（改写/扩展/意图分类）
优点	缺点
对口语化、模糊查询效果提升显著	每次多一次 LLM 调用，增加延迟（+0.5-2s）
同义词扩展对中文检索帮助大	改写可能引入歧义，导致检索偏离
意图分类可针对性选择检索策略	需要额外评估确保改写质量
7. 轻量级 Reranking（LLM-as-reranker 或小型 cross-encoder）
优点	缺点
排序质量大幅提升，特别是长尾复杂问题	LLM-as-reranker 额外增加一次 LLM 调用
LLM-as-reranker 零额外模型依赖	小型 cross-encoder 需要下载模型（~100MB）
可作为当前无法加载大型 reranker 的替代方案	reranking 结果与原始排序差异大时，需要调试
8. 检索自评估 + 失败重试
优点	缺点
低成本实现"伪 Agent"能力，无需完整重构	多轮重试增加延迟
处理检索为空或检索质量差的场景	需要定义"检索失败"的判断标准
可自动触发查询改写，形成反馈闭环	实现复杂度中等
9. 上下文长度管理
优点	缺点
防止 context 超出 LLM 窗口（静默失败或截断）	动态截断可能丢失关键信息
工作量小，确定性强	不同模型 context window 差异大，需要维护映射表
🔵 P3 类：长期/架构类
10. 语义分块（Semantic Chunking）
优点	缺点
分块质量从根本上提升，影响所有下游	需要在分块时做 embedding 计算，处理速度慢 2-5 倍
解决跨句子边界信息被割裂的问题	现有向量库需要重新 ingest
尤其对学术论文效果好	实现复杂，难以调试
11. Agent 重建（真正的 ReAct 循环）
优点	缺点
多步推理、动态工具选择、自适应能力	工作量大，需要从零设计
能处理当前 RAG 完全无法胜任的复杂任务	延迟高（多轮 LLM 调用）
与现有工具库（hybrid_search, calculate 等）契合	测试难度极高，幻觉风险更大
—	当前 LLM 推理能力（本地 Ollama 小模型）可能不足以支撑复杂 Agent 决策

需要跨多步推理（先查A，根据A的结论再查B）
需要使用外部工具（计算、联网、执行代码）

三、方案选择建议
场景	建议
想快速看到效果	只做第一、二阶段（约 1-2 周），RAG 质量可从"基本可用"提升到"较好"
知识库主要是中文文档	优先做 jieba 分词 + 查询扩展，收益远大于其他
知识库文档量大（>10k 文档）	BM25 缓存修复是必须的，否则搜索会非常慢
用户提问较复杂（多步、对比）	优先做查询预处理 + 轻量 reranking
需要真正的 Agent 能力	评估是否值得，建议用 LangGraph 从零搭建，不要在当前架构上打补丁
