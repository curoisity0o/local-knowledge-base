# RAG 知识库系统优化说明

> 本文档记录本次优化涉及的所有改动、设计思路和使用方式。

---

## 一、RAG 检索能力提升

### 1.1 当前架构与不足

| 能力 | 优化前 | 优化后 |
|------|--------|--------|
| 检索方式 | 纯向量相似度搜索 | 混合搜索（向量 + BM25）+ 父子上下文 + 多查询 |
| 上下文窗口 | 固定 chunk 大小（800字）| 父块提供更完整上下文（1500字） |
| 查询变体 | 单一查询 | 可选多查询变体（Multi-Query） |
| 检索策略选择 | 硬编码 | 可通过配置灵活切换 |

### 1.2 新增检索策略

#### ① 混合搜索（Hybrid Search）——已有，Agent 侧已接入
- 将稠密向量检索（语义相关）与 BM25 稀疏检索（关键词精确匹配）的结果用 **RRF（Reciprocal Rank Fusion）** 融合
- 适合大多数场景，是推荐的默认检索方式
- `vector_store.hybrid_search(query, k=5)`

#### ② 父子上下文检索（Parent-Child Retrieval）——**新增**
- 文档在入库时同时生成小粒度子块（400字）和大粒度父块（1500字）
- 向量存储索引**子块**（精准匹配），检索时命中子块后扩展返回**父块内容**（提供完整上下文）
- 对需要前后文理解的问题效果显著（如"上文提到的…"、跨段落关联等）
- 通过 `vector_store.parent_child_search(query, k=4)` 调用

```yaml
# config/settings.yaml
document_processing:
  chunking:
    use_parent_child: true      # 启用父子分块摄入
    parent_chunk_size: 1500
    child_chunk_size: 400

rag:
  retriever:
    parent_child:
      enabled: true             # 启用父子上下文检索
```

#### ③ 多查询检索（Multi-Query Retrieval）——**新增**
- 使用 LLM 为原始查询生成 N 个不同角度的变体（默认 3 个）
- 分别对每个变体执行检索，用 RRF 融合结果去重排序
- 提升复杂语义问题的召回率（不同表述命中不同文档）
- 通过 `vector_store.multi_query_search(query, k, llm_manager)` 调用

```yaml
rag:
  retriever:
    multi_query:
      enabled: true
      num_queries: 3
```

### 1.3 检索策略优先级

`RAGChain._retrieve_documents()` 按以下优先级选择策略（只用一种）：

```
父子上下文 > 多查询 > 跨语言混合 > 普通混合 > 带阈值相似度 > 基础相似度
```

## 二、Chunk 分块方案优化

### 2.1 修复的问题

**`ChineseTextSplitter`（修复前）**：
- 分隔符列表存在重复项（`"****"` 出现 3 次，`"**"` 出现 2 次，`"__"` 出现 2 次）
- Markdown 格式标记（`**`、`__`、`##`）混入中文分隔符，造成切割位置混乱

**`MarkdownAwareSplitter.text_splitter`（修复前）**：
- 分隔符含 `"##"`、`"###"`、`"#"`，与 `MarkdownHeaderTextSplitter` 的标题切割冲突

### 2.2 优化后的分隔符策略

所有分块器统一使用清晰的优先级层次：

```
段落（\n\n → \n）> 代码/公式块（``` → $$）> 句子（。？！；? ! ;）
> 省略号（…… → …）> 词语（，, 空格）> 字符（空串兜底）
```

### 2.3 新增：父子分块器（`ParentChildSplitter`）

```python
from src.core.document_processor import ParentChildSplitter

splitter = ParentChildSplitter(
    parent_chunk_size=1500,  # 父块：完整上下文
    child_chunk_size=400,    # 子块：精准检索单元
    chunk_overlap=50,
)
child_docs = splitter.split_documents(documents)
# 每个子块的 metadata 包含：
# - parent_id: 归属父块的唯一标识
# - parent_content: 父块完整文本（检索时扩展使用）
# - chunk_index: 在父块内的序号
# - total_chunks: 父块包含的子块总数
# - parent_chunk_index: 父块在文档中的序号
```

**启用方式**（全局开关）：

```yaml
document_processing:
  chunking:
    use_parent_child: true
```

## 三、Agent 能力增强

### 3.1 原有问题

- `RAGAgent._retrieve_context()` 只调用 `similarity_search`，未使用已有的混合搜索
- 工具集单一：只有搜索、检索、列举文档三种工具
- 没有复杂问题的分解能力，遇到多跳问题容易漏检

### 3.2 新增工具（`src/agents/tools.py`）

| 工具名 | 说明 | 适用场景 |
|--------|------|----------|
| `hybrid_search` | 混合搜索（向量+BM25），推荐优先使用 | 绝大多数查询 |
| `parent_context_search` | 父子上下文检索，返回更完整上下文 | 需要前后文的问题 |
| `decompose_query` | 将复杂问题拆分为子问题 | 多跳/多方面问题 |
| `knowledge_summary` | 对指定主题生成知识库综合摘要 | 了解知识库内容 |

### 3.3 新增 Agent 类型

#### `MultiStepRAGAgent`（新增）
位于 `src/agents/rag_agent.py`，继承自 `RAGAgent`。

**特点**：
1. 自动识别复杂查询（字数 > 30 或含比较/对比等关键词）
2. 对复杂查询调用 `decompose_query` 拆分为 2-4 个子问题
3. 逐一对子问题执行混合检索，汇总上下文后生成完整答案
4. 内置迭代上限保护（max_iterations=8）

**使用**：
```python
from src.agents.rag_agent import create_multi_step_agent

agent = create_multi_step_agent(llm_manager=llm, vector_store=vs)
result = agent.process("比较文档A和文档B在XXX方面的异同")
```

#### `RAGAgent`（升级）
- `_retrieve_context()` 现在优先使用 `hybrid_search`，自动降级到 `similarity_search`
- `_init_tools()` 注册了 `hybrid_search` 和 `parent_context_search` 两个新工具

### 3.4 `get_default_tools()` 接口更新

```python
# 旧接口
registry = get_default_tools(vector_store=vs)

# 新接口（增加 llm_manager 参数，用于 decompose_query / knowledge_summary）
registry = get_default_tools(vector_store=vs, llm_manager=llm)
```

## 四、前端替代方案：Gradio

### 4.1 为什么提供 Gradio 替代

| 维度 | Streamlit | Gradio |
|------|-----------|--------|
| 代码量 | ~1150 行（含大量状态管理） | ~280 行 |
| 状态管理 | 手动维护 `session_state` | 框架内置，自动 |
| 对话组件 | 需要自行实现气泡样式 | `gr.Chatbot` 原生支持 |
| 文件上传 | 需要手动处理文件对象 | `gr.File` 直接返回路径 |
| API 挂载 | 独立进程，需 CORS 配置 | 可内嵌为 FastAPI 子路径 |
| 实时流式 | 需要 `st.empty()` + 手动刷新 | 内置 streaming 支持 |

### 4.2 启动方式

**独立启动**：
```bash
# 确保 FastAPI 后端已在 8000 端口运行
python src/frontend/gradio_app.py --port 7860 --api-url http://localhost:8000
```

**挂载到 FastAPI**（可选，在 `src/api/main.py` 中添加）：
```python
import gradio as gr
from src.frontend.gradio_app import build_interface

gradio_app = build_interface()
app = gr.mount_gradio_app(app, gradio_app, path="/gradio")
# 访问：http://localhost:8000/gradio
```

### 4.3 功能覆盖

Gradio 前端实现了 Streamlit 前端的核心功能：

- ✅ 智能问答（多轮对话、来源引用）
- ✅ 检索参数调节（top_k、LLM 提供者、RAG 开关）
- ✅ 文档上传与处理
- ✅ 文档列表查看与删除
- ✅ 系统状态监控

## 五、配置变更汇总

### `config/settings.yaml` 新增字段

```yaml
document_processing:
  chunking:
    use_parent_child: false        # 是否使用父子分块（默认关）
    parent_chunk_size: 1500        # 父块大小
    child_chunk_size: 400          # 子块大小

rag:
  retriever:
    parent_child:
      enabled: false               # 是否使用父子上下文检索
    multi_query:
      enabled: false               # 是否使用多查询检索
      num_queries: 3               # 查询变体数量

agents:
  default_agent_type: "rag_agent"  # 可选: rag_agent | multi_step_rag_agent | research_agent
  multi_step_agent:
    max_iterations: 8
    complexity_threshold: 30
  tools:
    enabled_tools:
      - "hybrid_search"
      - "parent_context_search"
      - "decompose_query"
      - "knowledge_summary"
      # ... 其他工具
```

## 六、推荐配置组合

### 场景 A：快速验证（默认配置）
- 混合搜索 + 普通分块，开箱即用

### 场景 B：需要更好上下文理解的文档知识库
```yaml
document_processing.chunking.use_parent_child: true
rag.retriever.parent_child.enabled: true
```

### 场景 C：需要处理复杂多跳问题
```yaml
agents.default_agent_type: "multi_step_rag_agent"
# 或在代码中使用 create_multi_step_agent()
```

### 场景 D：提升召回率（牺牲一定速度）
```yaml
rag.retriever.multi_query.enabled: true
rag.retriever.multi_query.num_queries: 3
rag.retriever.top_k: 10
```
